# 每日新闻项目经验与收获

## 一、产品与时间窗设计

- 两次推送要覆盖完整 24 小时，不能只按“当天自然日”机械切片。
- 目前更合理的时间窗方案是：
  - 早间：前一天 17:00 → 当天 08:00
  - 晚间：当天 08:00 → 当天 17:00
- 这样既保留了每天两次推送节奏，也不会漏掉 17:00-24:00 的新闻。

## 二、中文综合热点抓取的关键决策

- 中文 `general` 通道不能再依赖猜测式 RSS 入口。
- 新华、人民网、央视、澎湃、界面等看似存在的 RSS 入口，实测并不适合生产主线。
- 对中文综合热点，真正有效的路线是：
  1. 权威媒体栏目页 / 列表页抽取
  2. 详情页二次抓取标题、摘要、发布时间
  3. 搜索引擎作为补漏层，而不是主采集层

## 三、质量控制的实际经验

- `general` 通道最难的不是“抓不到新闻”，而是“抓到了太多不适合日报正文的内容”。
- 央视和界面都接上后，真正值钱的是：
  - 事件级去重
  - 题材过滤
  - 来源权重平衡
- 界面这种源不能只做宽泛关键词白名单，否则会把证券、融资、公司深稿重新放回来。
- 更有效的做法是：
  - 栏目黑名单强拦
  - 更窄的题材白名单
  - 央视权威稿优先保留，界面作为补充

## 四、调试与可观测性经验

- `selected=0` 这类问题不能靠猜，要按阶段打日志。
- 事实证明，`general` 一度归零并不是抓取坏了，而是同日 state 库去重把结果全挡掉了。
- 因此专门增加了：
  - `--ignore-state` 联调开关
  - `.debug.md` 分阶段日志
- 这使得问题可以被明确拆分为：
  - 抓取层
  - 题材过滤层
  - 去重层
  - state 层

## 五、飞书知识库自动化经验

- 先把“能写进去”打通，再优化“写得好不好看”，这条顺序是对的。
- 飞书文档的自动写入不要直接把 markdown 当纯文本逐行塞进去，否则呈现会很糙。
- 更好的做法是把最小 markdown 结构映射成 docx block：
  - `#` → 一级标题
  - `##` → 二级标题
  - `1.` → 有序列表
  - `-` → 无序列表
  - 其他正文 → 普通段落
- 飞书 docx blocks API 在大正文写入时必须批量化，否则容易触发 429 限流。

## 六、生产化经验

- “更新已有日报文档”在飞书 docx 场景里并不稳，尤其容易碰到 block 删除 / 404 问题。
- 更工程化的方案是：
  - 首次跑生成标准标题文档
  - 当天重复跑时，自动创建 `（重跑 HHMMSS）` 版本
- 这样更适合真实生产场景：
  - 可重跑
  - 可补跑
  - 可追溯
  - 不覆盖旧版本

## 七、项目当前收获

- 这次项目已经从“脚本原型”推进到“可生产运行的自动化链路”：
  - 抓取
  - 去重
  - 飞书知识库落库
  - 自动推送链接
  - cron 定时执行
- 更重要的收获不是代码量，而是把几个关键判断跑清楚了：
  - 中文综合热点不能再赌 RSS
  - 调试必须可观测
  - 生产环境优先选择稳定策略，而不是优雅但脆弱的实现
