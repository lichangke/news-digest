# 每日新闻项目｜收尾与落地 TODO

> 文档目的：
> 这不是继续为了重构而重构的计划，而是一份面向当前 `news-digest` 项目的正式收尾清单。
> 目标是在现有主链路已经打通的前提下，把项目从“阶段性可用”推进到“稳定可维护、可长期运行”。

---

## 一、当前项目判断

当前 `news-digest` 的状态，不再是“是否能做出来”，而是：

> **主链路已经成立，但还需要完成最后一轮工程化收束。**

从现有设计、验证和运行文档综合判断：

- 抓取 → 去重 → 渲染 → 写入知识库 → 通知，这条链路已经具备闭环能力
- `general` / `ai` 双通道结构已经建立
- `runs/*.json`、`.debug.md` 等可观测性能力已经成为正式结构的一部分
- 当前项目已经具备继续稳定运行的基础

但与此同时，仍有几项关键工作没有完全收口：

- 主流程入口仍需进一步统一
- `general` 通道质量控制仍需继续打磨
- 部分质量规则仍与抓取逻辑耦合
- 失败恢复路径和运维手册还需要补齐

因此，接下来的重点不是继续大规模重构，而是：

> **先统一骨架、再验证真实运行、再收紧策略与运维。**

---

## 二、收尾原则

在执行以下 TODO 时，遵循 4 条原则：

1. **产出优先于优雅**
   - 日报正常生成、归档、可补发，优先于代码形式上的完美

2. **真实运行优先于 demo 结论**
   - 是否可维护，应以真实抓取、真实写入、真实通知为准，而不是以本地 demo 为准

3. **结构收束优先于功能扩张**
   - 当前阶段不应继续盲目扩源、加新特性，而应先把已有主链路收稳

4. **观察驱动后续重构**
   - 只有当真实高频摩擦点出现时，才继续推进下一轮结构性调整

---

## 三、按执行顺序拆分的 TODO 清单

# Phase 1：先把主骨架钉死

## 1. 核查并收紧唯一标准入口约定

### 当前判断
这一项**已基本完成**。

当前代码现实是：
- `app.pipeline` 已经是标准主流程入口
- `scripts/run_morning.sh` / `scripts/run_evening.sh` 都直接调用 `python -m app.pipeline`
- `app.main` 仍然存在，但已经退化为 legacy 兼容壳层，内部委托给 `app.pipeline`

因此，这一项不再是“从零开始统一入口”，而是：

> **确认约定已落地，并继续防止未来重新出现平行入口。**

### 待办
- [ ] 盘点当前所有启动方式（`app.main`、shell 脚本、README 示例、cron 调用点）是否都已指向 `app.pipeline`
- [ ] 检查是否仍存在遗留的旧调用路径或文档示例
- [ ] 确认 `app.main` 只保留兼容壳层职责，不再承载独立主流程逻辑
- [ ] 在维护文档中继续强调：后续新增能力只能接入 `app.pipeline`

### 完成标准
- 无论手动运行、定时任务、联调模式还是补跑，底层都进入 `app.pipeline`
- `app.main` 不再演化为第二条事实主流程
- 未来新增逻辑时，不再需要同时修改多条执行路径

### 说明
这一项的风险已经明显下降，但仍值得保留为“制度化核查项”。
真正要防的是未来维护过程中重新长出新的平行入口。 

---

## 2. 核查并收紧运行产物与状态边界

### 当前判断
这一项也**已基本形成现实分工**，但仍需要文档与代码层面的最终收口。

当前代码现实是：
- `runs/*.json` 已经是单次运行事实记录的主来源
- `scripts/latest_run.sh`、`scripts/retry_notify.sh` 等维护动作都围绕 `runs/*.json` 工作
- SQLite 当前真正被主流程使用的是 `articles` 表，用于去重状态判断与写回
- SQLite 中虽然创建了 `runs` 表，但当前主流程并未把它作为运行事实主源使用

因此，这一项不再是“从零开始统一边界”，而更像是：

> **把已经形成的事实分工正式制度化，并清理掉未来可能造成误解的残留结构。**

### 待办
- [ ] 在文档中明确：`runs/*.json` 是单次运行事实记录主来源
- [ ] 在文档中明确：SQLite 主要承担 state 去重职责，当前不作为主运行事实来源
- [ ] 核查是否仍存在遗留的旧逻辑会把运行事实写入 DB 并参与主流程判断
- [ ] 评估是否需要保留 SQLite 中未被实际使用的 `runs` 表；若保留，需写清其非主职责定位
- [ ] 在维护文档中写清 `runs`、`debug`、`state` 三者的排障顺序与职责边界

### 完成标准
- 排障时具备明确顺序：
  1. 先看 `runs/*.json`
  2. 再看 `.debug.md`
  3. 再看 shell / cron 日志
  4. 最后再看 SQLite state
- 不会因为 SQLite 中残留的 `runs` 表定义，误导维护者以为 DB 和 JSON 是并列真相源

### 说明
这一项当前最大的风险，不是分工尚未出现，而是：

- 文档如果写得过于“未完成”，会低估当前已经形成的稳定约定
- 代码里若继续保留未使用的 `runs` 表定义，又可能让未来维护者误以为系统存在两套并列运行记录机制

所以这里真正要做的，是**收口与防误解**，而不是重新设计。

# Phase 4：把“失败后能恢复”做扎实

## 7. 收紧并验证失败恢复路径

### 当前判断
这一项**已部分完成**，而且基础能力已经真实存在。

当前代码与脚本现实是：
- notify 失败后，已经可以通过 `scripts/retry_notify.sh` 基于已有 `runs/*.json` 补发
- `runs/*.json` 已按阶段记录 `wiki_sync` / `notify` 结果，具备基础失败隔离能力
- shell 脚本会把 notify 结果回写到 `runs/*.json`
- 飞书文档重跑时，当前已有“生成重跑版本而非覆盖原文档”的策略实现

因此，这一项不再是“从零开始建立失败恢复路径”，而是：

> **把已存在的恢复能力验证清楚、约定清楚，并补上仍未制度化的边界。**

### 待办
- [ ] 验证通知失败后补发路径在真实场景下稳定可用
- [ ] 验证 wiki 写入失败后，维护者能否仅依赖现有文档与运行产物快速定位问题
- [ ] 明确单通道失败时另一通道是否继续产出，并将这一行为写入文档约定
- [ ] 明确哪些情况应补发，哪些情况应整条链路重跑
- [ ] 确认“重跑生成新文档”策略在真实场景下表现稳定，不会破坏历史结果

### 完成标准
- 系统不是依赖“永不失败”运行，而是即使失败也不会进入混乱状态
- 维护者知道每类故障应该如何处理
- 已存在的失败恢复能力被明确文档化，而不是只存在于脚本行为里

### 说明
生产系统的关键不是没有异常，而是异常发生后仍可恢复、可追溯、可补救。

这一项当前剩下的重点，已经不是“有没有恢复机制”，而是：
- 恢复机制是否在真实场景下稳定可信
- 文档约定是否足够清楚，能让维护者不靠猜测处理故障

也就是说：
- 第一阶段目标已经达成：有一份可用的最小运维手册
